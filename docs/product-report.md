# 订阅账本 — 产品报告

## 一、产品定位

一个本地优先的订阅管理工具。帮用户回答两个问题：

1. **我这个月花了多少订阅费？**
2. **下一个该续费的是什么？什么时候到期？**

不碰银行对接，不做自动扣款检测。用户手动录入或粘贴导入，数据全部存本地。

---

## 二、核心数据模型

```
AppState
├── budget: number        ← 月预算（手动设）
├── rows: SubscriptionRow[]  ← 订阅列表
│   ├── id, category, plan, fee
│   ├── subscribed: boolean  ← 是否已订阅
│   ├── dueDate: string      ← 续费日（YYYY-MM-DD）
│   ├── subscribedAt: string ← 订阅日期
│   ├── expired: boolean     ← 手动标记过期
│   └── usage: string        ← 备注（订单号、激活码等）
└── bills: Bill[]           ← 账单流水
    ├── subscriptionId      ← 关联到哪条订阅
    ├── amount, paidAt
    ├── orderId             ← 订单号
    └── kind: "payment" | "renewal"
```

**逻辑关系：**
- **订阅**是"我有什么服务" — 按月续费的周期性订阅
- **账单**是"我实际付了多少钱" — 每次付款的流水记录
- 月支出按**账单付款日**统计，不按订阅月费估算
- 续费操作会自动生成一条 `kind: "renewal"` 的账单

## 三、现有功能盘点

### 保留的核心功能

| 功能 | 说明 | 评价 |
|------|------|------|
| **订阅列表** | 表格展示所有订阅，按到期紧急度排序 | ✅ 核心，保留 |
| **账单流水** | 按时间排列的付款记录，可搜索 | ✅ 核心，保留 |
| **预算追踪** | 本月支出 vs 月预算进度条 | ✅ 核心，保留 |
| **待续费提醒** | 3天内到期的订阅，可一键续费 | ✅ 核心，保留 |
| **统计** | 按分类支出 + 近6个月趋势 | ✅ 核心，保留 |
| **服务库** | 预置 60+ 常见订阅，一键添加 | ✅ 有用，保留 |
|| **粘贴入账** | 粘贴订单文本/JSON，自动解析成账单 | ✅ 核心入口，保留 |

### 已清理的功能

| 功能 | 删除原因 |
|------|----------|
| HTML 迁移 | 从浏览器版迁移的一次性功能，已过时 |
| 导出/导入 JSON | 依赖文件系统对话框，偏离"纯本地录入"流程，已移除 |
| 旧版 acceptance test | 依赖 HTML 迁移逻辑，随主功能删除 |

## 四、当前导入能力分析

现在**只有一条导入路径** — "粘贴入账"，支持 3 种格式：

| 格式 | 输入 | 解析逻辑 | 问题 |
|------|------|----------|------|
| **中转订单文本** | `订单 HS123 · 2026-07-05 · ¥140` | 正则提取订单号、日期、金额 | ⚠️ 只认特定格式 |
| **账单 JSON** | v3 备份或 bills 数组 | JSON.parse 直接读 | ✅ 可靠 |
| **OpenAI 用量 JSON** | `{ data: [{ amount, date }] }` | 读 data/usage 数组 | ⚠️ 只支持 OpenAI 格式 |

### 关键短板

1. **不支持图片** — 截图、账单照片无法识别
2. **不支持邮件** — 不能从邮件正文提取账单
3. **不支持短信** — 不能从短信提取扣款通知
4. **格式识别太死** — 中转站文本格式一变就解析不了
5. **账单关联太粗** — 导入的账单默认挂到最后一条订阅，经常关联错

## 五、产品优化方向

### 第一优先级：多源导入（老大明确要求）

**目标：** 图片、文字、邮件、短信都能导入订阅账单。

#### 1. 图片导入（OCR + 智能解析）

```
用户操作：粘贴截图 / 选择图片文件
    ↓
OCR 提取文字
    ↓
智能解析（识别金额、日期、订单号、服务名）
    ↓
预览确认 → 写入账单
```

**实现路径：**
- **方案 A（推荐）：** 用系统级 OCR（macOS Vision Framework / Tauri 调用）提取文字，再用现有的文本解析逻辑处理。零 API 成本，离线可用。
- **方案 B：** 调用 AI 视觉模型（GPT-4o / Claude）直接从图片提取结构化账单数据。更准但有 API 成本。
- **建议：** 先做 A，解析不了时 fallback 到 B。

#### 2. 邮件导入

```
用户操作：粘贴邮件正文 / 上传 .eml 文件
    ↓
提取邮件正文（去 HTML 标签）
    ↓
识别扣款关键词（"订阅"/"续费"/"账单"/"订单"/"付款"）
    ↓
提取金额、日期、服务名、订单号
    ↓
预览确认 → 写入账单
```

**实现路径：**
- 先支持**粘贴邮件正文**（最简单，复用文本解析）
- 再支持 **.eml 文件解析**（用 mailparser 库）
- 不做 IMAP 自动拉取（太重，偏离"简单"原则）

#### 3. 短信导入

```
用户操作：粘贴短信内容
    ↓
识别扣款通知模板
    ↓
提取金额、日期、商户名
    ↓
预览确认 → 写入账单
```

**短信常见格式：**
- `【支付宝】您于2026-07-05向XX科技支付140.00元`
- `【银行】您的信用卡于07月05日扣款140元`

**实现路径：** 纯正则 + 模板匹配，不需要 AI。

#### 4. 统一导入入口

现在"粘贴入账"只有文本框。改成统一的导入面板：

```
┌─────────────────────────────────┐
│  导入账单                         │
│                                 │
│  [粘贴文字] [粘贴图片] [选择文件]  │
│                                 │
│  ┌─────────────────────────┐    │
│  │  文本框 / 图片预览区      │    │
│  │                         │    │
│  └─────────────────────────┘    │
│                                 │
│  识别结果：                       │
│  ✓ 金额 ¥140  ✓ 日期 2026-07-05  │
│  ✓ 订单号 HS123  ⚠ 服务名未识别  │
│  关联到：[示例订阅 ▼]             │
│                                 │
│  [取消]          [确认导入]       │
└─────────────────────────────────┘
```

**关键设计：**
- 自动识别格式（文本/JSON/图片/邮件），不需要用户手动选
- **必须预览确认**，不直接写入
- 识别不到的字段让用户手动补

---

### 第二优先级：现有功能打磨

#### 1. 账单自动关联

现在导入的账单默认挂到最后一条订阅，经常关联错。

**改法：** 导入时按服务名/订单号自动匹配已有订阅，匹配不到时让用户手选。

#### 2. 续费日批量检查

现在续费日到期后，需要手动点"已续费"才会推到下个月。

**改法：** 到期后自动弹通知（已有 tray 通知基础），点击通知直接续费。

#### 3. 订阅去重

从不同渠道导入可能产生重复订阅。

**改法：** 按 `plan + category` 去重，已有逻辑但只在粘贴导入时触发，应扩展到所有导入路径。

---

### 第三优先级：不做的功能

| 功能 | 不做原因 |
|------|----------|
| 银行/支付平台自动对接 | 偏离"本地私密"定位，技术复杂度太高 |
| 多用户/云同步 | 违背本地优先原则，增加维护成本 |
| 自动汇率换算 | 增加复杂度，用户自己换算更简单 |
| 订阅推荐/比价 | 偏离工具定位，变成内容产品 |
| 复杂图表/数据看板 | 现有统计够用，过度可视化是冗余 |

## 六、技术架构建议

```
packages/core/          ← 纯逻辑，不依赖 UI
├── types.ts            ← 数据模型
├── catalog/            ← 服务库 + 导入解析器
│   ├── connectors.ts   ← 现有文本/JSON 解析
│   └── parsers/        ← 新增：按格式拆分
│       ├── ocr.ts      ← 图片 OCR 解析
│       ├── email.ts    ← 邮件正文解析
│       ├── sms.ts      ← 短信解析
│       └── text.ts     ← 通用文本解析（现有逻辑迁移）
└── ...

apps/desktop/           ← Tauri + React
├── ImportPanel.tsx     ← 统一导入面板（替代 PasteSyncModal）
└── ...
```

**原则：**
- 解析逻辑全部在 core 层，UI 只负责展示和确认
- 每种格式一个 parser，互不依赖
- 新增格式只需加一个 parser 文件 + 注册，不改现有代码

## 八、一句话总结

**现在的产品核心是好的 — 本地、私密、简单。** 要做的不是加功能，是把"导入"这条路打通：让用户不管从哪里拿到账单信息（截图、邮件、短信、文本），都能 3 步内导入并确认。其他功能不碰。
