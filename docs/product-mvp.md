# 订阅账本 · 产品 MVP 说明

> 定位：**只做订阅 / 会员 / 周期扣费统计**，不做网购与日常记账。  
> 本文档为范围与对外说法；**具体价格在公测前再定**。

---

## 1. 产品一句话

只管订阅和会员，不管日常网购；AI、影音、设计工具和中转站，能记清、能提醒、能算每月实付与下一笔续费。

---

## 2. 目标用户（Primary Persona）

内测与文案默认面向以下 **1～2 类主用户**（可并列，不追求全民记账）：

| Persona | 特征 | 典型订阅 |
|---------|------|----------|
| **创作者 / 独立开发者** | 同时用多家官方 AI + 中转 + 额度包；订单号、激活码多 | AI 订阅、中转服务、开发工具、各类 API 额度 |
| **小团队负责人（可选）** | 要汇总团队 SaaS 支出，不想上企业费控与银行绑卡 | 协作工具、设计、AI 席位 |

**非目标用户**：只订 1～2 个流媒体、且从不记订单的人（表格 / 系统日历够用）。

传播时可收窄为：**「创作者 AI 与工具订阅账本」**（中文案例 + 模板），比泛「订阅管理」更易在社群转化。

---

## 3. 边界（必须遵守）

### 做

- 软件 / 服务 **订阅、会员、周期扣费**
- **额度包、中转站**（按「订阅条目」管理，非按量云账单全量）
- 订阅相关 **账单**（付款日、金额、订单号、续费记录）
- **预算与统计**（仅订阅支出）
- **续费提醒**（应用内为主；桌面通知视端能力）

### 不做

- 淘宝、外卖、转账等 **日常消费记账**
- **银行卡全流水**同步与消费自动分类
- 报销、家庭总账本、理财

## 5. MVP 范围

### 5.1 必须做（替代现有 HTML 账本 + 值得装 App）

| 模块 | 说明 |
|------|------|
| 订阅条目 | 名称、分类、金额、续费日、已订阅 / 未订阅 / 已过期；额度包按「非周期」处理（可无续费日） |
| 状态与视图 | 概览、**已过期**、**账单**、**待续费**（默认 3 天内到期 + Tab 角标） |
| 统计 | **本月支出**（见 §6 口径）、月预算、预算条、**下一笔续费** |
| 账单 | 关联订阅、付款日、金额、订单号、备注；**一键续费**推进 `dueDate` 并写入 `kind: renewal` 账单 |
| 提醒 | 到期前 N 天（默认 3 天）；**应用内**待续费列表；桌面/系统通知仅在壳或前台支持时启用，不 over-promise |
| 数据 | 导入/导出 JSON（兼容 `ai-subscription-tracker-v3`）、本机存储（Phase 0：SQLite） |
| 迁移兼容 | 自 v3 导入；可选从备注解析历史订单进账单（内部迁移逻辑，不对外承诺「自动识单」） |
| 端 | 优先桌面或 Web+桌面壳；手机 Phase 2 |

### 5.2 业务规则（与 HTML v3 对齐，写入 Core）

- **额度包 / 赠额**：`isCreditLike` 类条目不要求续费日，不参与「待续费」与按日提醒。  
- **未订阅**：视为愿望清单，清空续费日，不计入有效订阅数。  
- **过期**：手动 `expired` 或 `dueDate` 已过 → 进「已过期」，**不计入**下一续费排序（本月支出仍只跟账单走）。  
- **续费日推进**：周期默认按月 `+1`（与现 HTML `nextMonthlyDueDate` 一致）；年付等扩展放 MVP 后。

### 5.3 MVP+1（MVP 稳定后）

- 每条：`subscribeUrl` / `portalUrl`（一键打开续费页）
- **手动同步（连接器 v0）**：仅 **1～2 种**已验证的面板格式或「粘贴 JSON/响应」；对外称「手动同步」，不写「通用 URL 全自动」
- 预置服务卡片（省输入，不承诺全自动）

### 5.4 后续（不写入首版对外承诺）

- 多设备云同步
- 邮件订单解析（仅订阅类商户）
- 官方 OAuth / API
- 连接器远程更新

### 5.5 Phase 0 验收标准

- 导入同一份 v3 JSON 后：**本月支出、预算剩余、下一续费、待续费条数与名称** 与 core demo/parity 摘要一致。  
- 导出 JSON → 再导入 **往返无损**（`id`、账单条数、`kind` 保留）。  
- 离线可用；无网络时增删改查与提醒逻辑正常。  
- 续费操作：改 `dueDate` + 新增 renewal 账单，统计立即更新。

## 6. 统计与预算口径（对外一句话）

| 指标 | 规则 |
|------|------|
| **本月支出** | 仅统计 **`bills[].paidAt` 落在当月** 的金额合计；**不是**有效订阅 `fee` 简单加总。 |
| **月预算 / 预算条** | 对比「本月支出」与用户设置的月预算。 |
| **年付 / 大额额度包** | MVP：**仅在付款月计入本月支出**（不自动摊销到 12 个月）。若未来做「预估固定月支出」视图，与「本月实付」分开展示。 |
| **下一续费** | 有效订阅、非额度包、有 `dueDate` 的最近一条。 |

商店与帮助页建议原句：**「本月花了多少，按你记账的付款日算；没记账单就不计入当月。」**

## 7. 数据模型（与现有 HTML 对齐）

### Subscription（对应 `rows[]`）

| 字段 | 说明 |
|------|------|
| `id` | 稳定 ID |
| `category` | 展示分类；见下表映射 |
| `segment`（可选） | 规范化：`ai` \| `media` \| `design` \| `office` \| `relay` \| `credit` \| `other` |
| `plan` | 套餐 / 产品名 |
| `fee` | 月费或展示用金额（字符串可含「/月」等，解析时剥离） |
| `subscribed` | 是否已订阅 |
| `dueDate` | 续费日 `YYYY-MM-DD` |
| `subscribedAt` | 开通日 |
| `expired` | 手动过期标记 |
| `usage` | 备注（订单号、激活码等） |
| `subscribeUrl` | （MVP+1）订阅/面板链接 |
| `portalUrl` | （MVP+1）续费页 |

### 分类：v3 HTML → 新客户端

| v3 `category`（HTML） | 建议 `segment` | 说明 |
|----------------------|----------------|------|
| 官方 | `ai`（默认）或按 `plan` 细分为 media/design/office | 导入时默认 `ai` |
| 中转 | `relay` | |
| 中转额度包 | `credit` | 非周期、无待续费 |
| 其他 | `other` | |
| 用户自定义字符串 | 保留 `category` 展示，segment 填 `other` 或规则推断 | |

新客户端 UI 可用 segment 驱动图标/筛选，**导入 v3 时必须保留原 `category` 文案**。

### Bill（对应 `bills[]`）

| 字段 | 说明 |
|------|------|
| `id`, `subscriptionId`, `amount`, `paidAt`, `orderId`, `note` | 已有 |
| `kind` | `payment` \| `renewal` |

### 存储键（迁移）

- 当前 HTML：`ai-subscription-tracker-v3`
- 新客户端：支持导入 v3 整包 `{ budget, rows, bills }`；字段经 `normalizeRow` / `normalizeBill` 兼容（见 HTML 实现）

## 8. 对外文案（无价格）

### 商店 / 官网短描述

**标题**：订阅账本 — 只记会员与订阅，不记日常消费  

**正文要点**：

1. **订阅专用**：月费、年付、额度包分开看；预算只看软件与服务实付。  
2. **创作者友好**：官方 AI、中转、额度包同一套清单；续费日与账单在一起。  
3. **数据在你这边**：本机优先，支持导出备份。（云同步上线后再补充说明。）

**边界（建议显眼）**：

> 我们不记录淘宝、外卖等日常消费，也不连接银行流水；只帮助管理「订阅与会员」支出。

### 提醒能力（诚实表述）

- **可以写**：到期前提醒、待续费列表、打开应用时检查。  
- **Phase 0 慎写**：「后台永远推送」——需 Phase 2 移动端或系统级调度。  

### 不要现在写

- 具体价格、终身买断价  
- 「自动支持一切软件 / 一切中转面板」

### 可以写

- 「逐步增加可手动同步的订阅类型」  
- 「暂不支持的订阅也可手动记录，一样参与统计与提醒」

## 9. 定价策略（原则，非价目表）

| 阶段 | 动作 |
|------|------|
| 内测 0 | 免费，换反馈（最常记的 10 类订阅） |
| 内测 1 | 免费，看 7 日留存、是否维护 10+ 条、是否打开「待续费」 |
| 公测前 | 定免费版上限与付费项（见下「默认假设」） |
| 公测后 | 按付费意愿与竞品现价定区间；可试买断 vs 年付 |

**内测必问**（完整卷见 [beta-survey-and-analytics.md](./beta-survey-and-analytics.md)）：

1. 愿意为「纯订阅」单独用 App，还是 Notion/表格够用？  
2. 最可能付费的能力：中转同步、多设备同步，还是统计与提醒？  
3. 大约几条订阅？（影响免费条数上限）  
4. 每月订阅类支出大概多少？（&lt;200 / 200–800 / &gt;800 元）

**公测前「免费版」默认假设（可内测调整）**：

| 免费 | 付费解锁（候选） |
|------|------------------|
| ≤15 条订阅、本机、导入导出 | 无限条、多设备同步、连接器、高级导出/快照 |

付费锚点优先级建议：**多设备同步 &gt; 连接器 &gt; 条数**；提醒单独高价售卖意愿通常较弱。

## 11. 市场备忘（对内，非对外）

- **需求真实但垂直**：AI 时代多人 5～15 个工具订阅；整体更像 prosumer 工具，非大众记账。  
- **替代品强**：Excel/Notion 免费；Bobby 等一次性买断。需靠 **中转/AI 场景 + 待续费闭环 + 本机信任** 换付费。  
- **风险**：连接器维护成本；一旦宣传「自动同步」会被持续期待。策略：**窄承诺、手动为主**。  
- **机会**：中文创作者社群、桌面工作流、隐私敏感用户；内容向传播（「我每月 AI 订阅读多少」）优于纯功能堆砌。
